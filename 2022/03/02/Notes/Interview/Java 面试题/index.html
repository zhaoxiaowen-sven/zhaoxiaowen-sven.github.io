面经_02Java 面试题

一、HashMap

数据结构

  • JDK1.8：数组 + 链表 + 红黑树，红黑树：提高查询的效率（ <= 2logN-1），降低插入和删除的效率

  • JDK1.7：数组 + 链表

扩容是2的倍数？

  1. 计算节点的数组索引index，位移的效率比取余高
  2. 扩容时计算index，hashcode & （n-1）只要看高位就能确定扩容后的index
  3. 2^n -1 高位和低位都能参与计算，hash值分布均匀

扩容因子 0.75？

  • 空间利用率高，hash冲突概率低

树化阈值8?

  • 泊松分布，出现概率低 百万分之6

转链表阈值6?

  • 避免红黑树和链表的频繁转化

Hash规则(1.8)

hash = hashCode ^ hashCode >>> 16

数组索引计算 hash * （n-1），n 是数组长度

为了数据分布更加均匀，在下一步计算 节点 的数组索引。

简述扩容过程？

  1. 计算新数组大小，扩容阈值
  2. 创建新数组
  3. 数据迁移 ，将oldTable 数据迁移到newTable，index 有2种情况，1oldIndex ，newIndex = index + oldCap ，移动节点
  4. 红黑树转链表（可能）（1.8）

hashMap 并发的安全问题？

  • 1.7 链表节点死循环，头擦法
  • 1.8 元素覆盖，尾插法

二、ConcurrentHashMap

数据结构

  • JDK 1.7 ：Segment 数组 + HashEntry数组 + 链表Segment[HashEntry[cap] ssize]

    参数说明：

    1. concurrencyLevel ：Segment 数组长度（并发阈值，影响 segmentShift，segmentMask计算

    2. 容量大小 initialCapacity <= cap * ssize，否则的话 cap * 2。

  • JDK 1.8：数组 + 链表 + 红黑树

Hash 计算规则?

JDK1.7:

下边的hash值是通过哈希运算后的hash值，不是hashCode

  1. 计算 Segment 下标 = (hash >>> segmentShift) & segmentMask

    计算 Segment 数组下标是用的 hash值高几位（这里以高 4 位为例）和掩码做与运算，

  2. 计算 HashEntry 数组下标:(tab.length - 1) & hash

计算 HashEntry 数组下标是直接用的 hash 值和数组长度减1做与运算。

这是为了尽量避免当前 hash 值计算出来的 Segment 数组下标和计算出来的 HashEntry 数组下标趋于相同。
简单说，就是为了避免分配到同一个 Segment 中的元素扎堆现象，即避免它们都被分配到同一条链表上，导致链表过长。同时，也是为了减少并发。

JDK1.8：和HashMap 相同

并发设计?

  • 1.7：ReentrantLock，基于Segment的分段锁设计
  • 1.8：CAS + synchronized 锁头节点

扩容规则?

  • JDK1.7：找到 lastRun 节点，最末尾的几个连续的节点，他们的hash值相同整体移动， 剩余节点从头开始依次复制，头插法插入到新的链表中。

  • JDK1.8：扩容规则：找到 lastRun 节点，从它到尾结点整体迁移，lastRun前边的节点则单个迁移，头插法插入到新的链表中。有一点和1.7不同，1.7 lastRun前边的节点是复制过去的，而这里是直接迁移的，没有复制操作。

三、JUC

volatile

特性：

  • 保证可见性
  • 保证有序性，禁止指令重排 -> 单例 半初始化问题
  • 不保证原子性（需要借助synchronized或者CAS)

底层实现：lock指令

锁

分类：

  • 乐观 （CAS）+ 悲观（synchronized）

    是否锁定资源

  • 自旋锁 VS 适应性自旋锁

    如果对于某个锁，自旋很少成功获得过，那在以后尝试获取这个锁时将可能省略掉自旋过程，直接阻塞线程，避免浪费处理器资源。

  • 公平 + 非公平：是否按照顺序获取锁

  • 重入 + 非可重入：同一个线程多次重入

  • 独占锁 + 共享锁:

CAS（compare And swap）

  1. 原理：比较和交换（原子性），cas 的原子性是由底层CPU指令x86(cmpxchg)支持的

    UnSafe类（C 实现）

  2. 应用：Atomic XXX

  3. 问题：

    1. 循环时间长开销大
    2. 对象原子性操作：AtomicStampedReference
    3. ABA 问题（AtomicStampedReference，添加版本号）

  4. 自旋：不断的进行CAS 操作，通常会配置自旋的次数，防止死循环。

synchronized

  • 原子性
  • 可见性
  • 有序性

Java 中每个Object 都拥有一把锁，存放在对象头中。

对象内存布局

8 bytes的倍数，方便CPU读取。

  • 对象头：KlassPointer + 数组长度 + markWord （锁状态 + 分代年龄，垃圾回收）

    markWord （8字节）

  • 数据

  • 对齐

锁状态

  • 无锁：没有加锁，

  • 偏向锁：只有一个线程访问资源且没有竞争。

    markWord 中记录着偏向线程的id

  • 轻量级锁：多个线程访问锁单没有竞争。

    指向Java 栈中锁记录的指针，复制对象头中markRecord到 Java栈中LockRecord，并且生成双向指针。其他线程自旋等待

  • 重量级锁：Object Monitor

    1. 对象的实现：字节码monitorenter/monitorexit 底层实现 ObjectMonitor 对象， entrySet + owner+ waitSet
    2. 方法的实现：字节码中（ACC_SYNCHRONIZED）

volatile vs synchronized

volatile synchronized
修饰变量 修饰方法 代码段
可见性/不保证原子性 可见性+原子性+有序性
不会造成线程阻塞 会造成线程阻塞

synchrionized VS ReentrantLock

synchrionized ReentrantLock
公平性 非公平 公平/非公平
支持中断 否 是， lockInterruptibly
等待队列 1个 多个Condition队列
实现原理 ObjectMonitor(JVM) aqs + volatile(JDK)

AQS

抽象队列同步器，JUC 包的基石，ReentrantLock + ReentrantReadWriteLock + Countdownlaunch + Semaphore等都是基于AQS的；包括线程池 Worker也是继承AQS。

核心思想：如果被请求的共享资源空闲，那么就将当前请求资源的线程设置为有效的工作线程，将共享资源设置为锁定状态；如果共享资源被占用，就需要一定的阻塞等待唤醒机制来保证锁分配。

核心原理：

  1. 互斥变量的设计和如何保证互斥变量的线程安全性？

    通过volatile + cas。ivolatile state值表示锁状态，>0 表示被锁定，=0表示无锁。一个线程获取锁时，如果state=0 ，会被更新为1，表示被占用。

    当多个线程同时获取锁时，为保证共享变量更新的原子性，需要CAS机制更新。

  2. 线程阻塞和唤醒

    多个线程同时获取锁时，没有获取到锁的线程会被阻塞，在锁被释放后再进行竞争。

    线程阻塞和唤醒基于的是同步队列，同步队列是一个FIFO双向队列，当出现资源竞争需要阻塞时，就会将竞争的线程包装成一个Node节点，放入同步队列中。同时会调用UnSafe#park将线程阻塞；当锁被释放后，在通过UnSafe#unpark 会唤醒同步队列中的下一个线程节点，再次获取锁。

  3. 公平和非公平性

    AQS在获取锁时，公平锁会去判断当前同步队列是不是为空，如果不为空，则需要加入同步队列进行等待；

    而非公平锁会不管同步队列中是否有阻塞的线程，都先通过cas获取一下锁，如果获取到了就先执行，否则就加入同步队列。

  4. 独占共享：

    ReentrantLock 是独占锁，state>0 时判断当前的持有锁线程和要获取锁的线程是不是一个

    ReentrantReadWriteLock ：读锁是共享锁获取读锁的时候会判断写锁的状态，如有写锁没有被持有

    写锁是排他锁

    共享锁：CountDownlaunch + Semaphore

    中断的原理：AQS中，检测到中断时，是否抛异常，自旋获取锁的方法中

  5. 超时的原理：UnSafe类park超时返回

  6. 等待唤醒机制：condition 队列，await或notify时线程已经获取到锁了。

    await：释放Node节点的锁，挂起节点线程（当前线程），创建新节点加入到条件队列中

    notify：将条件队列中的节点移动到同步队列中，唤醒节点线程竞争锁。

ThreadLocal

每个Thread 维护一个 ThreadLocalMap 映射表，这个映射表的 key是 ThreadLocal 实例本身，value 是真正需要存储的 Object。
Thread#ThreadLocalMap<Entry<WeakReference<ThreadLocal>, Object>>
内存泄露：
Threadlocal是弱引用对象，当发生GC时，会被置位null，ThreadLocal中Entry对象的key变为null，但是Object是强引用，不能被回收。
当线程生命周期没有被终结的时候，object 就不能被回收。

  1. 使用完成后，手动调用下remove方法，清除数据。
  2. set 、get方法中也会清除key 为null的数据
  3. 使用线程池的情况下，没有及时清理ThreadLocal，可能导致业务逻辑异常。
